MTPRO backend security & architecture review

Demo 基线具备可验证的边界；生产下单路径不应获批，直到库级生产出口被硬性封死。

MTPRO 已形成清晰的 DomainModel、MessageBus、RiskEngine、OMS、ExecutionClient 与只读 Dashboard 分层。当前 CLI 和 GitHub Demo workflow 均锁定 Binance Demo，但公开的 canary library 仍能构造生产 host 请求，且多个 API 默认环境为 .production。这使“默认关闭”停留在集成约束，而不是库级不可绕过的安全属性。

公开 canary library 仍可向生产 Binance host 发出签名下单请求。

Security control assessment

控制有效性

以下评估区分“默认值、测试或工作流约束”与“任何 caller 都无法绕过的库级约束”。

Effective

证据路径 containment

读取时拒绝绝对/相对逃逸、symlink、root 外路径及读取前后 canonical mismatch；缺失或损坏一律 blocked。

Effective

默认依赖拒绝操作

Canary transport 的 credential provider、network loader 和 artifact sink 默认均拒绝，未注入时不可发起 I/O。

Effective

Demo workflow 隔离

workflow_dispatch、binance-demo environment、Demo secrets、固定 Demo CLI 和上传前红脱敏检查形成一条受控运行路径。

Ineffective for production

生产禁止只是状态字段

runner 最终 evidence 始终写入 false，但没有把 request.environment 限制为 Demo，也没有拒绝 production endpoint。

Over-privileged

PR CI token 权限过宽

写权限位于整个 workflow 的顶层，而非仅发布 job；检查脚本本身没有写仓库的业务需要。

Needs hardening

审批无独立信任根

approval packet 是本地 JSON，identity 取自环境变量；需要独立审批主体、签名和服务器侧验证才能用于任何生产权限。

Remediation sequence

建议的修复路径

按风险和可验证性排序。完成前不应把 v0.33.x 的 Demo 证据提升为生产 execution readiness。

P0 · 生产出口硬阻断

将 v0.33 canary 类型限制为 Demo

移除或 internalize .production endpoint、默认值和 production host allowlist；在 runner 与 transport 双层显式 guard environment == .demo。未来生产实现应放在单独 target/feature gate，并默认不编译进 Demo runtime。

Exit gate任何公开 v0.33 canary API 对 production host 都返回确定性错误；负向测试覆盖 direct library caller。

P1 · CI 最小权限

下收 contents: write

把 workflow 顶层改为 contents: read；仅 release-publication job 声明写权限。审查所有 action pin 与 self-hosted runner 的 trust boundary，PR job 不接触发布 credential。

Exit gatepull_request workflow 的有效 token 不能推送、创建 release 或修改仓库内容。

P2 · 授权可信度

用独立审批替代自述 JSON

将审批绑定到 GitHub Environment reviewer、OIDC/签名的 approval record 或独立审批服务；要求 approver 与 operator 不同，记录签名、commit、scope、额度、有效期和单次 nonce。

Exit gate篡改 JSON、同人自批、过期、跨 commit、跨产品或跨 symbol 均被拒绝。

P2 · 凭据卫生

增加忽略与提交前扫描

忽略 .env*、*.pem、*.key、credential/evidence 临时目录；在 PR fast lane 加 gitleaks 或等价 secret scan，并保留已跟踪路径的 allowlist。

Exit gate测试模拟 key、私钥标记和非空 credential 文件都会令 CI 失败。

P3 · 复杂度收口

按运行时职责拆分大文件与历史 guard

优先拆分 live boundary、canary/evidence persistence、risk gate 与 session store；将历史文档锚点检查分离到 contract test，保持业务行为测试面向 public API。

Exit gate每个拆分后的模块有独立所有者、目标依赖检查和 focused behavior test。

保持现有优势

不要破坏已验证的 Demo 防护

保留 endpoint allowlist、redaction、artifact checksum、run lock、Demo workflow environment 和 Dashboard read-model-only 边界。修复生产出口时，不得扩大 DataClient、Dashboard 或策略层权限。

Exit gateDemo submit/status/cancel 和所有 redaction / path-escape 负向测试继续通过。

Production authorization gate

生产授权的最低验收门槛

当前报告不建议发起生产 cutover。以下内容是未来独立生产项目的最小验收集合，而不是本次 Demo 审计的完成项。

Code

不可绕过的 capability gate

  • production transport 位于独立 target
  • 默认构造器不含 production host
  • 每个入口双层拒绝未授权环境
  • library caller 负向测试覆盖

Identity

独立审批与凭据

  • operator 与 approver 职责分离
  • 审批签名绑定 commit/scope/limit
  • 短期、最小权限凭据
  • secret 不进参数、日志、文件或 artifact

Runtime

受控执行语义

  • symbol、notional、leverage 严格 allowlist
  • 风险、kill switch、no-trade 为实时权威输入
  • 幂等键与持久 run lock
  • submit/status/cancel/reconcile/rollback 完整闭环

Evidence

可验证、可审计证据

  • provenance 由受信来源产生
  • artifact 只保存 redacted 数据
  • checksum、realpath、权限与 retention 可验证
  • 独立审计能重放拒绝与失败路径

Delivery

供应链与发布

  • PR token 只读，发布 job 最小写权限
  • secret scan、依赖审计、SBOM
  • macOS/Linux 完整矩阵与 Dashboard smoke
  • 回滚路径和事故演练有证据

Decision

审批语义

  • 测试通过不等于生产可用
  • Demo 证据不等于 production authorization
  • 每一次开通有独立 change record
  • 任一关键证据缺失即 fail closed
