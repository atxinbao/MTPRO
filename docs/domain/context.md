Date: 2026-07-20

## Purpose

This document defines the current shared language used by MTPRO code, reviews,
operator documentation, and validation. It is not an execution plan and does not
authorize a release, production cutover, credential read, endpoint connection,
or order.

The former 1,400-line issue-anchor ledger is preserved at:

`docs/history/architecture-pre-canonicalization-2026-07-20/context.md`

Historical terminology in that snapshot describes completed stages; it is not
the current product definition.

## Strategy Language

- EMA and RSI are the active concrete strategies.
- A strategy emits a signal or target exposure intent, not an exchange order.
- Spot and USD-M Futures product rules are explicit; Spot cannot inherit
  Futures-only short/leverage semantics.
- Strategies do not read credentials, choose production endpoints, call
  ExecutionClient, or mutate OMS.
- New strategies require a separate product/strategy contract and validation.

## Data and Execution Language

| Term | Meaning |
| --- | --- |
| Market event | Normalized DomainModel fact produced from DataClient input |
| Strategy signal | Research/runtime observation emitted by EMA or RSI |
| Target exposure intent | Strategy-owned desired exposure, before risk |
| Risk decision | Allow/reject/block result generated by RiskEngine |
| Order intent | Risk-approved internal execution request |
| OMS state | Internal order lifecycle truth managed by ExecutionEngine |
| Execution request | Venue-specific request prepared by ExecutionClient |
| Execution evidence | Redacted submit/status/cancel result linked to run/order identity |
| Reconciliation | Comparison and repair evidence between internal and external observed state |
| Rollback evidence | Proof that a canary or incident path can be stopped/recovered |

Canonical path:

```text
Market event
-> MessageBus
-> Trader strategy intent
-> RiskEngine decision
-> ExecutionEngine / OMS
-> ExecutionClient
-> Binance
-> execution evidence
-> MessageBus / Database / Portfolio
-> CLI / Dashboard
```

No component may skip the risk, OMS, evidence, or authorization stages.

## Runtime Modes

| Mode | Purpose | External side effects |
| --- | --- | --- |
| Research / Backtest | Deterministic historical evaluation | None |
| Paper | Local simulated execution | None |
| Demo | Binance Demo Network validation | Demo-only, explicitly configured |
| Production shadow/read-only | Production-shaped observation without order mutation | Read-only and separately gated |
| Production live | Real order mutation | Separate cutover authorization required |

All modes target the same DomainModel, MessageBus, strategy, risk, OMS, and
Portfolio semantics. Environment-specific behavior belongs in clock, data,
adapter, credential, endpoint, and authorization implementations.

## Evidence Language

| Term | Meaning |
| --- | --- |
| runID | Identity shared by one runtime/evidence chain |
| correlationID | Identity linking related events/commands |
| causationID | Identity of the immediate cause |
| artifact manifest | Namespaced list of evidence files and checksums |
| immutable evidence bundle | Validated artifact set that is not rewritten after acceptance |
| trusted provenance | Repository/workflow/run/actor facts obtained from the trusted source, not caller booleans |

## Current Production Boundary

Production is not permanently forbidden, but it is currently not authorized.

The following statements are mandatory:

```text
productionCutoverAuthorized=false
defaultProductionTradingEnabled=false
```

Demo credentials, Demo success, a CLI flag, Dashboard input, documentation
change, maintenance PR, or release tag cannot implicitly enable Production.

## Forbidden Interpretations

- Binance Demo parity means Production is enabled.
- DataClient can submit/cancel/replace orders.
- ExecutionClient can generate strategy intent.
- Dashboard can create production authorization.
- Cache is durable truth.
- Portfolio owns raw broker payloads.
- A historical planning record is an active queue.
- A published release is still "pending publication".
- A retired module name remains an active source root.
- OKX or Bybit is part of the current product scope.
